AUGUST 15, 2020

Saturday

Change Character Color Implemented

Today Kip and I were able to get the prototype of the character customization screen working. Last week we struggled with being able to dynamically change the image of the character using the menu buttons using the require function. This week we decided to use a style property called ‘tintColor’ to change the color of the blob instead of actually changing the image of the blob for each color. We struggled a bit with the syntax of changing the state, but eventually we were able to get this working. TintColor changes the color by applying the specified color on all the nontransparent pixels of the image, so since our initial ‘blob’ is white, whatever color we specify in tintColor will appear. This works for our prototype, but eventually if we want to add patterns or other customization beyond color changing, we may run into issues and will need to change our implementation. For now, we have a functioning prototype/proof of concept of our character customization screen. :)

We also started working on displaying the xp in the progress bar of our top navigation bar. Initially, I thought this would be relatively simple. Since the xp value is just stored in Async storage and the progress bar can change its appearance based on whatever value between 0 and 1 is passed to it, I thought we could just pull the current xp out of Async storage, divide it by the xp needed to pass the level, and pass that decimal value into the progress bar. However, because of the way that promises work with Async storage, this was not nearly as simple as I anticipated. Using promise.then() works to manipulate the values, but it cannot return the value as a number, it can only return promises, which cannot be used by our progress bar. If we try to use other methods to resolve the promise value and just return the value stored in the promise, then when the xp is updated in Async storage, the resolved value will not change to reflect this update, so our progress bar won’t update in real time. Kip and I struggled with a few different, but were not able to come up with a solution.

This is our last week working on the app for this summer, so we might just need to make a note of where we’ve left off and what we’ve tried and then continue working on this issue whenever we pick up on the app. Alternatively, if our TaskList team finishes early tomorrow, we might be able to work on the top navigation together. Although our progress bar isn’t working, I’m glad that we were able to get our character customization prototype working before the end of the summer.

We did this by using the promise.then() function. Before calling promise.then(), we resolved the value from GetXP so that we could use the up-to-date value of the XP. NaN Troubles

This morning Kip, Allen, and I started our task of incrementing XP when users check off a task. Our approach was to use Async storage in a similar way to how we saved userID a few weeks ago. This approach allows multiple components of the app to access the same XP value and allows the XP to stay stored so that it doesn’t get reset when the user exits out of the app. However, we ran into issues with object types. Async storage only stores string values, but XP should be a numeric value since we need to increment and decrement the value. We tried to store it as a string in Async storage and just convert it to a number when we need to increment and decrement the value, then convert it back to a string when we save the value back to storage. However, we kept getting NaN values instead of the values we were expecting. We tried testing different approaches for where we were converting values and we added print statements throughout our code to see where the error was coming from. It seems like the error is coming from the step where we save the xp, but we aren’t sure how to proceed. We seem to be stuck this week, but we will continue to work through this problem next week.

Later, we had our weekly team meeting. We started the meeting by talking through some of the app logistics. We confirmed some of the concepts we discussed last week, like how users’ XP is tracked by the progress bar at the top of each screen, then when the XP reaches a set amount where the progress bar is full, the next level will automatically check off and the user will get an alert that the level has been unlocked and the progress bar will zero out again. Once the level is checked off, the user can navigate to the quest screen to collect the new item. The item will only be added to the user’s inventory when they click on the quest and collect the item. We talked through some additional details, like if a user has already passed a level, but then they go back and uncheck a task, they will have negative XP in the progress bar instead of reverting the level and removing the item. Ultimately, the user still has to collect the same amount of XP to unlock the next level in either situation. But the negative XP method seems more simplistic and more intuitive. We also decided that the level numbers should be split by chapter, so it will be level 1.9 or level 2.6 instead of just continuously increasing the level number regardless of the chapter number (Ex: if chapter 1 has 10 levels, the 11th level would be level 2.1 instead of level 11).

Lastly, we caught up on this week’s progress and discussed next week’s plans. Allen and Wayne were able to get a modal popup working when users click on a level, but they are having issues with formatting, specifically with rendering the bitmoji at the correct size. Next week they will fix these formatting issues with Bitmoji. Kip, Allen, and I updated the team on our struggles with Async storage, which we will continue working on next week. Kip and I talked about our attempts at rendering a customizable Bitmoji using Libmoji (see yesterday’s post) which we will continue working on next week. JULY 24, 2020

Friday

Struggling With Libmoji Traits

Today Kip and I struggled with Libmoji for the duration of our work session. A consistent problem with Libmoji seems to be the lack of documentation. The aspect of Libmoji we are working with is the traits, which allows users to customize their character’s basic traits like hair, skin color, and facial features. The problem we are running into is with the formatting. Libmoji has a function that pulls the traits called getTraits(). The problem is that the input to this function has to be a Trait object, but we have no way to see what a Trait object should really look like or how to initialize a Trait object. We tried printing a Trait object using the given Libmoji randTrait() function, but it just printed a Javascript array. When we tried printing the type of a trait object, it just returned ‘object’ which was too generic to learn any additional information from. We tried all sorts of syntactic variations on Javascript arrays by moving brackets and commas into different grouping, but nothing seemed to work. The possible values for different traits and their customizations are stored in a JSON file. So, our next approach was to try different methods of pulling values directly from the JSON file, and then transforming those values to Javascript, leaving it as JSON, etc. None of these attempts yielded any successful results. Eventually, we just used a long URL template that was provided to render the Bitmoji. While this works for rendering in a specific Bitmoji, it doesn’t seem very scalable or intuitive for allowing users to customize their character. Kip and I will have to look more into how we can change the URL implementation or keep working on formatting to pull the traits from the JSON file so that it’s more scalable for customization.
