JULY 3, 2020

Friday

Fixing Async Storage

Today Kip, Allen, Wayne, and I met up to work through some issues with Async storage. Allen and Wayne have been primarily working with this issue, so they started off the meeting by sharing what they had tried and what they were having issues with. Next, we compiled some articles that walked through how to use Async storage with React Native and tried following along with these tutorials. We ran into quite a few obstacles along the way, but ultimately, with some tweaks, new dependencies, and new files we were able to figure out how to use Async storage to save a user ID. Hopefully this will help Allen and Wayne with their Login screen progress, and we can apply the same ideas of Async storage to other aspects of the app like coins, quests, characters, etc. so that our app works in the way that we want.

The last step was to run some tests to make sure we were correctly saving the userID and that we would be able to retrieve the userID when needed. We had some trouble with this portion initially since AsyncStorage returns objects as promises, so we were not able to see the userID as a string. Eventually, using await and .getItem(), we were able to create an alert that welcomes the user to the app using their userID! We probably won’t have this alert in the final app, but this is a good way to test that the userID is stored and retrievable.

Today I also realized that it might be useful to apply some of the project management strategies that I’ve learned through my Fidelity internship to this project. In my internship, my team uses a backlog to keep track of features to implement during their Agile sprints. Our app project doesn’t use sprints, but I think that organizing a backlog of features that have yet to be implemented could be helpful. The backlog is organized according to the MoSCoW Prioritization, which has 4 categories: Must Have, Should Have, Could Have, Won’t Have. Must Haves are items that are vital in order for the product to be considered successful/functional. For our app, these would be the minimum features that we would need for our prototype by the end of the summer. Should Haves are items that are high-priority but not essential. So for our app these would be the items that make our app unique and would be the first things we would work on adding to the prototype, but are not essential for functionality. Could Haves are items that would be nice or are desirable, but are not necessary for the app. These features might be included in the first updates of our app. Won’t Haves are items that were brainstormed but may end up being rejected because they aren’t feasible or no longer align with the vision of our app. I’m planning to bring up MoSCoW prioritization during Sunday’s meeting and seeing what the rest of the team thinks.

Fixing Buttons

Last night, Kip was able to fix the dependency issues with the packages by reinstalling all the necessary packages. :) With these issues fixed, we were able to get working on what we had set out to work during our last meeting. Our aim for this session was to move the check box to the left of the task text and implement a button to allow users to delete tasks. This setup is similar to how other apps allow users to add and delete items, so hopefully this setup will make it more intuitive for users.

The first step was to move the checkbox from the left side of the text to the right side of the text. This proved to be a bit more challenging than we originally anticipated. The first, most straightforward step to move the position of the checkbox’s code to be before the text’s code. Since React Native renders items left to right, this would move the checkbox item to the left of the text as desired. However, this led to some problems with flexbox widths. Somehow, the boxes had expanded to the full width of the screen, which meant that the checkbox and the text were left-aligned on separate lines. It took quite a while of experimenting with flexboxes and different parameters like flex-wrap, justifyContent, flex-direction, align-items, align-content, etc. Eventually, we were able to create appropriately sized boxes by hard-coding widths and putting an additional container around the text boxes to control the size of it’s flexbox without affecting the layout of the text. With the correctly sized text boxes, we were able to move the checkbox button and the text onto the same line using flex-direction.

Our next step was to create a button that allows users to delete tasks from their list using an ‘X’ button. The tutorial that we worked off of last week included a function for a delete button, so we were able to re-implement this button pretty quickly and easily. Lastly, we translated our hardcoded widths to flex values in order to make our layout more flexible to different sizes of screens without running into formatting issues. We also quickly made some aesthetic changes regarding colors and spacing.

We discovered that the edit feature was no longer functioning after we made our changes today. This will be our first priority for next week since it's an essential part of our Task list’s functionality. I’m not quite sure what happened between last week and this week, but we’ll look more into this next week. Unfortunately, we weren’t able to implement the feature where completed tasks move to the bottom of the list automatically or allow users to move the items manually. Swipeable Navigation

A short navbar update entry for today. Kip and I worked on implementing the swipe feature to the bottom navigation bar for about an hour and talked through some items for the top naviagtion bar. We worked based on the tutorial found last Saturday and though there were a couple obstacles, it was pretty straightforward!

Initially, we had implemented the navigation bar using createBottomTabNavigation (from two weeks ago). This allowed us to create a clickable bottom navigation bar, but did not allow for swiping. We tried to see if we could implement swiping using creatBottomTabNavigation or even with createMaterialBottomTabNavigation for a while, but did not have any luck. It turns out that in order to have a swipeable bottom navigation bar, we needed to use createMaterialTopTabNavigation and specify that placement should be at the bottom. Admittedly, this seems a little messy, but createMaterialBottomTabNavigation does not allow for swiping. :( The ‘Material’ aspect of the createMaterialTopTabNavigation also allows for the animation of sliding screens when swiping. With the swiping functionality enabled, users still have the functionality to click the buttons on the bottom navigation bar to get to different screens. Once functional, to keep the navigation bar clean, we decided to remove the labels and only display icons on the navigation bar. Since there are only 3 screens and we will display the name at the top of each screen, we felt that the labels were unnecessary. After some quick formatting changes, we finished up our new swipeable bottom navigation bar. :)

Divide And Conquer

We had our second team meeting today! Today’s meeting was pretty short, but we made some important decisions about next steps and did a lot of designation of tasks during this meeting. We started our meeting by discussing where we were with learning React so that we could gauge how comfortable people were with React. The team will follow the pair-programming model of a driver (write code) and a navigator (look out for errors) in order to write code. This set-up allows us to work collaboratively on the components of the add without bumping into each other. :) We discussed how we wanted to split up the components, which was a bit difficult, since there are some parts of the app that we have not completely finished ideating. First Team Meeting!

We had our first team meeting today. We talked through our work breakdown structure in terms the basic components we would need to cover relating to defining our problem, understanding user needs, designing a system, prototyping, and project management activities. We have decided that we will hold meetings every Sunday once internships and research start for most of our team. We talked through the pros and cons of different languages for mobile app development. Ultimately, we have decided to build the basic prototype of our app in React since it works with both iOS and Android. If we feel constrained by React we will consider switching to Xcode or even C++ later on.

We also spent some time talking about our user group. Since our target age demographic is teens and 20s, we wanted to make sure we were appealing to this audience. We noticed that nostalgia for childhood is popular with this age group, so we may try and tie in the aesthetics from the 1990s and early 2000s (ex: Avatar, Animal Crossing, Club Penguin). For this age group representation and justice is very important, so we will make sure that our avatar customization features include representation options that are inclusive of groups that may not be well-represented in traditional media (POC and LGBTQ+). We also think that our storyline for our quests should include elements that are important to this age group, such as justice, free will, technology, etc.

Next, we discussed user interaction and what we felt was lacking from existing apps. We found that existing gamified task managers can be stressful in terms of “punishing” the user when they fail to complete tasks. Instead of punishment, we are thinking about encouraging “streaks” (keep completing tasks for consecutive days) by offering rewards.

Lastly, we went through our team contract just to make sure we are all on the same page and to have a written agreement in the event that we run into issues or disagreements later. We decided that before the next meeting, we should all get familiar with React. We found a couple of online tutorials and agreed to complete or attempt them before our next meeting on June 7.
